# Rename Subscription Types: source‚Üísource_release, project‚Üísemantic_release

## Context

The current subscription type values `"source"` and `"project"` are confusing ‚Äî they describe what entity you're subscribing to, not what you receive. Users expect to choose between "Source Releases" (raw release notifications) and "Semantic Releases" (AI-analyzed reports). Renaming to `source_release` / `semantic_release` makes the intent clear.

## Scope

Rename the subscription `type` column values across the full stack: DB schema + migration, Go handlers, Go store, Go tests, notification routing, agent orchestrator, frontend types/form/page, API docs.

**Old ‚Üí New:**
- `"source"` ‚Üí `"source_release"`
- `"project"` ‚Üí `"semantic_release"`

### Step 1: Update DB schema (`internal/db/migrations.go`)

Change the schema string:
- Line 92 comment: `source-level and project-level` ‚Üí `source releases and semantic releases`
- Line 96: `CHECK (type IN ('source', 'project'))` ‚Üí `CHECK (type IN ('source_release', 'semantic_release'))`
- Lines 102-103: `type = 'source'` ‚Üí `type = 'source_release'`, `type = 'project'` ‚Üí `type = 'semantic_release'`

Add a post-schema migration block in `RunMigrations` (after schema exec) to migrate existing data:
```sql
UPDATE subscriptions SET type = 'source_release' WHERE type = 'source';
UPDATE subscriptions SET type = 'semantic_release' WHERE type = 'project';
ALTER TABLE subscriptions DROP CONSTRAINT IF EXISTS subscriptions_type_check;
ALTER TABLE subscriptions ADD CONSTRAINT subscriptions_type_check CHECK (type IN ('source_release', 'semantic_release'));
```

Note: Since schema uses `CREATE TABLE IF NOT EXISTS`, existing tables won't have their CHECK updated by the schema. The ALTER statements handle this. The UPDATE + ALTER approach is safe because:
- Fresh DBs get the new CHECK from CREATE TABLE
- Existing DBs get data migrated + constraint swapped by ALTER
